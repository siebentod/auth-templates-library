# AGENTS.md

Этот файл описывает архитектуру проекта, договорённости и правила для AI-агентов, работающих с кодовой базой.

---

## Архитектура

Проект использует **слоистую архитектуру (Layered Architecture)**. Каждый слой имеет строго определённую ответственность и может обращаться только к слою ниже.

```
routes → controllers → services → repositories → database
              ↓
          middleware
              ↓
        shared (errors, utils)
```

### Правила зависимостей между слоями

- `routes` импортируют только `controllers` и `middleware`
- `controllers` импортируют только `services` и `shared/errors`
- `services` импортируют только `repositories` и `shared/*`
- `repositories` импортируют только `schemas` и `config/db`
- `middleware` импортируют `repositories`, `shared/*` и `config/*`
- `schemas` не импортируют ничего из `src/` — только внешние библиотеки

**Нарушение этих правил недопустимо.** Если нужно добавить зависимость, которая нарушает граф — это сигнал пересмотреть решение.

## Описание слоёв

### `config/`
Инфраструктурная инициализация. Содержит три файла и больше не должен расти.

- `env.ts` — единственный источник правды об окружении. Все остальные файлы берут переменные отсюда, **никогда напрямую из `process.env`**.
- `db.ts` — экземпляр Drizzle и pg Pool. Экспортирует `db`.
- `app.ts` — фабрика Express-приложения. Регистрирует middleware и роутер.

### `schemas/`
Drizzle-схемы таблиц. Единственный источник правды о структуре БД. Содержат только описание таблиц и экспорт инферированных типов (`$inferSelect`, `$inferInsert`). **Никакой логики.**

### `types/`
TypeScript-типы и DTO. Не содержат runtime-кода — только `type` и `interface`.

- `UserPublic` — модель пользователя **без поля `password`**. Это единственный тип, который возвращается в HTTP-ответах.
- `JwtPayload` — содержимое Access Token: `{ sub: string, role: Role }`.
- `express.d.ts` — расширение типа `Request`. Добавляет `req.user?: JwtPayload`.

### `shared/errors/`
Иерархия ошибок. Все бизнес-ошибки наследуются от `AppError`.

| Класс | HTTP-код | Когда использовать |
|---|---|---|
| `NotFoundError` | 404 | Ресурс не найден |
| `UnauthorizedError` | 401 | Нет или невалидный токен |
| `ForbiddenError` | 403 | Нет прав, аккаунт заблокирован |
| `ConflictError` | 409 | Дубликат уникального поля |
| `ValidationError` | 400 | Ошибка бизнес-валидации (не Zod) |

Сервисы и middleware **бросают** ошибки. Контроллеры **пробрасывают** их через `next(err)`. `errorHandler` **обрабатывает** их.

### `shared/utils/`
Stateless вспомогательные функции.

- `hash.ts` — `hashPassword`, `comparePassword` (bcrypt), `hashToken` (SHA-256 для refresh токенов).
- `jwt.ts` — `signAccessToken`, `verifyAccessToken`, `generateRefreshToken`, `generateTokenPair`, `refreshTokenExpiresAt`.

### `repositories/`
Единственный слой, который знает о Drizzle и SQL. Принимают примитивные аргументы, возвращают типизированные объекты.

**Правила для репозиториев:**
- Никакой бизнес-логики — только CRUD и простые фильтры.
- `userRepository` никогда не возвращает поле `password` наружу, кроме метода `findByEmailWithPassword` — он используется исключительно в `auth.service.ts` для сравнения пароля.
- Методы возвращают `null` если запись не найдена — никогда не бросают `NotFoundError`.

### `services/`
Слой бизнес-логики. Оркеструет репозитории, применяет правила доступа, бросает типизированные ошибки.

- Сервисы **не знают** об HTTP (`Request`, `Response`, статус-кодах).
- Сервисы **не импортируют** друг друга. Если нужна межсервисная логика — вынести в отдельный сервис.
- Каждый публичный метод сервиса — это один сценарий использования (use case).

### `middleware/`

- `authenticate.ts` — проверяет Access Token, дополнительно делает запрос в БД чтобы убедиться что пользователь существует и активен. Кладёт payload в `req.user`.
- `authorize.ts` — фабрика `authorize({ roles?, orOwner? })`. Вызывается **после** `authenticate`. Проверяет роль или совпадение `req.user.sub === req.params.id`.
- `validate.ts` — фабрика `validateBody(schema)`. Парсит `req.body` через Zod-схему, при ошибке отвечает `400` напрямую (не через `next`).
- `errorHandler.ts` — последний middleware в цепочке. Обрабатывает все ошибки. В `production` не раскрывает детали внутренних ошибок.

### `validators/`
Zod-схемы для валидации HTTP-запросов. Только описание схем и экспорт инферированных типов DTO. Используются в `routes` вместе с `validateBody`.

### `controllers/`
Тонкий HTTP-слой. Контроллер делает три вещи: извлекает данные из `req`, вызывает сервис, отправляет `res`. Вся обработка ошибок — через `next(err)`.

**Контроллеры не содержат:**
- Бизнес-логики
- Работы с БД
- Блоков `try/catch` с логикой (только проброс в `next`)

### `routes/`
Регистрация маршрутов и применение цепочек middleware. Никакой логики.

Порядок middleware в цепочке всегда:
```
validateBody? → authenticate? → authorize? → controller
```

## Модель безопасности

### Access Token
- Короткоживущий JWT (по умолчанию 15 минут).
- Подписан `JWT_ACCESS_SECRET`.
- Содержит `{ sub: userId, role }`.
- Передаётся в заголовке `Authorization: Bearer <token>`.
- При каждом запросе middleware дополнительно проверяет `isActive` пользователя в БД.

### Refresh Token
- Долгоживущий случайный opaque-токен (128 байт hex, `crypto.randomBytes(64)`).
- **Не является JWT** — не содержит payload, не может быть декодирован без БД.
- В БД хранится **только SHA-256 хэш** токена. Сам токен не хранится нигде.
- При использовании немедленно инвалидируется и заменяется новым (**rotation**).
- При блокировке пользователя все его refresh токены удаляются из БД.

### Защита от перечисления пользователей
`auth.service.login` возвращает одинаковую ошибку `"Invalid email or password"` и при несуществующем email, и при неверном пароле.

## Соглашения о коде

### Именование
- Файлы: `kebab-case` с суффиксом слоя — `user.service.ts`, `auth.controller.ts`.
- Классы и типы: `PascalCase`.
- Переменные и функции: `camelCase`.
- Колонки БД: `snake_case` (через маппинг Drizzle).

### Экспорты
- Репозитории и сервисы экспортируются как объекты-синглтоны: `export const userService = { ... }`.
- Middleware экспортируются как именованные функции: `export function authenticate(...)`.
- Фабрики middleware — как именованные функции, возвращающие middleware: `export function authorize(options) { return (req, res, next) => ... }`.

### Обработка ошибок
- **Репозитории:** возвращают `null` если не найдено, никогда не бросают `AppError`.
- **Сервисы:** бросают `AppError`-наследников, никогда не ловят их.
- **Контроллеры:** оборачивают вызов сервиса в `try/catch`, пробрасывают в `next(err)`.
- **`errorHandler`:** единственное место, где ошибки превращаются в HTTP-ответы.

### Типизация
- `strict: true` — обязательно.
- Запрещено использовать `any`. Допустимо `unknown` с последующим сужением типа.
- `req.user` проверяется на существование в контроллере перед использованием (несмотря на то что `authenticate` гарантирует его наличие — TypeScript этого не знает).

## База данных

### Миграции
- Ручные SQL-миграции хранятся в `src/db/migrations/`.
- Нумерация: `0001_init.sql`, `0002_add_field.sql` и т.д.
- Drizzle Kit используется для генерации (`db:generate`), но финальный контроль — за разработчиком.
- **Никогда не изменять** уже применённые миграции — только добавлять новые.

### Индексы
Таблица `refresh_tokens` имеет индексы по `token_hash` (основной lookup) и `user_id` (для удаления всех токенов пользователя). При добавлении новых запросов — оценить необходимость индексов.

## Добавление нового функционала

### Новый эндпоинт
1. Добавить Zod-схему в `validators/`
2. Добавить метод в `service`
3. Добавить метод в `controller`
4. Зарегистрировать маршрут в `routes/` с нужными middleware

### Новое поле в модели пользователя
1. Добавить колонку в `schemas/user.schema.ts`
2. Создать новую миграцию в `src/db/migrations/`
3. Обновить типы в `types/user.types.ts` при необходимости
4. Обновить Zod-схему в `validators/` если поле входит в запрос

### Новая роль
1. Обновить `pgEnum` в `schemas/user.schema.ts`
2. Обновить тип `Role` в `types/user.types.ts`
3. Добавить миграцию с `ALTER TYPE role ADD VALUE '...'`
4. Обновить логику в `authorize.ts` и/или сервисах по необходимости

## Что намеренно не включено

- **Rate limiting** — рекомендуется добавить `express-rate-limit` на `/auth/login` и `/auth/refresh` в production.
- **Helmet** — рекомендуется добавить для security headers.
- **Логирование** — рекомендуется Pino или Morgan для структурированных логов.
- **Тесты** — архитектура с явными зависимостями через аргументы функций позволяет легко подменять репозитории в unit-тестах сервисов.
- **Swagger** — типы и Zod-схемы являются основой для автогенерации OpenAPI-документации.
